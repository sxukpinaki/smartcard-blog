01 — Problem Statement

The Problem

In most educational institutions across India, canteen transactions are still handled through cash-based systems. Students queue up, fumble for change, and canteen staff manually track orders — a process that is slow, error-prone, and impossible to audit. This leads to long wait times during peak hours, cashflow mismanagement, difficulty in tracking spending, and zero accountability in transactions.

Beyond the operational inefficiency, cash systems offer no way to enforce spending limits, prevent fraud, or provide students and parents with transparent financial records. Lost cash cannot be recovered; overspending cannot be flagged; and the institution has no digital record of what was purchased or when.

Existing Solutions & Their Limitations

Several alternatives to cash-based canteen systems already exist in the market and literature:

Manual register-based attendance/payment systems are the most common in institutions, requiring no technology but being entirely dependent on human accuracy and honesty. Biometric systems using fingerprint or facial recognition offer stronger identity verification but raise significant privacy concerns, require costly specialised hardware, and break down under poor lighting or wet fingers. Mobile application-based payment systems such as UPI-based canteen apps require every student to own a smartphone and have an active internet connection — not always feasible for all demographics. QR code systems are cheaper but require individual scanning per transaction which is slow, and QR codes can be photographed and duplicated. None of these solutions offer a simple, low-cost, contactless, hardware-backed identity system that works offline and provides real-time cloud feedback.

Our Unique Solution

The Smart Card system uses RFID cards as student identifiers — the same technology used in metro cards, hotel key cards, and access control systems worldwide. Each student is issued a unique RFID card. When tapped at the canteen terminal, the ESP32 microcontroller reads the card UID, validates the student's profile and balance from the cloud, displays account details on the OLED screen, and allows the canteen seller to process a deduction or recharge through the web-based Seller Panel — all in under 5 seconds.

The system is contactless, low-cost, fast, auditable, and privacy-respecting. It avoids the biometric data concerns entirely, works even when the student has no phone, and provides a complete transaction trail stored securely in Firebase.

04 — Knowledge Acquisition

Training Phase 1

Before any hardware was connected or code written, a structured training phase was undertaken to build competency across all four technical domains required by the project. Each area maps directly to a subsystem of the final product.

ESP32 & Arduino IDE

Study began with the ESP32 microcontroller architecture — its dual-core Xtensa LX6 processor, GPIO layout, hardware SPI and I2C buses, and the built-in Wi-Fi stack. The Arduino IDE was configured with the ESP32 board support package. Foundational exercises included LED blink tests, digital input/output, and serial communication — confirming the development environment was correctly set up before integrating any peripheral.

MFRC522 RFID Integration

Training covered SPI protocol fundamentals, the MFRC522 register architecture, and practical use of the MFRC522 Arduino library. Exercises involved reading the UID from both RFID cards and key fobs, understanding card memory sectors, and implementing basic UID-matching logic. This formed the identity layer of the entire system.

OLED Display Programming

The Adafruit SSD1306 and Adafruit GFX libraries were studied through their documentation and example sketches. Training covered I2C address configuration (0x3C), rendering multi-line text at different font sizes, drawing geometric shapes, and dynamically updating display content — essential for the real-time balance and status feedback shown to canteen users.

Cloud Backend — Sheets & Firebase

Initial cloud training focused on Google Apps Script — designing a structured Google Sheets schema, creating doGet/doPost endpoint functions, and testing HTTP requests from the ESP32 using its built-in WiFiClientSecure library. As the project matured, training shifted to Firebase Realtime Database — covering JSON tree design, security rules, REST API authentication, and real-time data listeners in JavaScript web applications.

Test 1 — ESP32 LED Blink

Confirms the microcontroller is powered, Arduino IDE is correctly configured, and GPIO output works.

Algorithm
  1. Define GPIO 2 as the built-in LED pin
  2. In setup(): configure GPIO 2 as OUTPUT
  3. In loop(): set GPIO 2 HIGH (LED ON) and wait 1000ms
  4. Set GPIO 2 LOW (LED OFF) and wait 1000ms
  5. Repeat indefinitely — confirms ESP32 is executing firmware correctly

  pinMode(LED_PIN, OUTPUT);

  digitalWrite(LED_PIN, HIGH);  // LED ON
  delay(1000);
  digitalWrite(LED_PIN, LOW);   // LED OFF
  delay(1000);

Test 3 — RFID Reader

Validates SPI wiring on GPIO 18/19/23/21/22 and confirms MFRC522 library can read card UIDs.

Algorithm
  1. Include SPI and MFRC522 libraries; define SS_PIN=21, RST_PIN=22
  2. In setup(): initialise Serial at 115200 baud
  3. Start SPI bus with SPI.begin(SCK=18, MISO=19, MOSI=23, SS=21)
  4. Initialise MFRC522 with PCD_Init()
  5. Print "Scan RFID card..." to Serial Monitor
  6. In loop(): check if a new card is present using PICC_IsNewCardPresent()
  7. If card detected, read its serial with PICC_ReadCardSerial()
  8. Iterate through UID bytes and print each in hexadecimal format
  9. Halt the card with PICC_HaltA() to allow next scan
Arduino C++ · test_03_rfid.ino 
#include <SPI.h>
#include <MFRC522.h>

#define SS_PIN  21
#define RST_PIN 22

MFRC522 rfid(SS_PIN, RST_PIN);

void setup() {
  Serial.begin(115200);
  SPI.begin(18, 19, 23, 21);  // SCK, MISO, MOSI, SS
  rfid.PCD_Init();
  Serial.println("Scan RFID card...");
}

void loop() {
  if (!rfid.PICC_IsNewCardPresent()) return;
  if (!rfid.PICC_ReadCardSerial())  return;

  Serial.print("UID: ");
  for (byte i = 0; i < rfid.uid.size; i++) {
    Serial.print(rfid.uid.uidByte[i], HEX);
    Serial.print(" ");
  }
  Serial.println();
  rfid.PICC_HaltA();
}

06 — System Evolution

Version History

The Smart Card system evolved through five distinct versions, each driven by real limitations discovered during testing. This is the engineering narrative of how a simple RFID scanner became a full campus payment platform.

V1
Basic RFID + Google Sheets
PHASE 1 · WEEKS 1–4

The first working version connected the ESP32 to Google Sheets via a Google Apps Script REST endpoint. When a card was scanned, the ESP32 fetched the student's balance and displayed it on the OLED. The canteen operator entered the transaction amount via the Arduino Serial Monitor, and a second HTTP request deducted the amount from the sheet. All transaction records were appended to a Transactions sheet.

Limitations Found
  • Amount entry via Serial Monitor is impractical in a real canteen
  • No web interface — operator must be at a PC with Arduino IDE open
  • Google Sheets polling is slow (~2–3 second delay)
  • No email receipt for students
  • No seller authentication
What Was Achieved
  • Full RFID → ESP32 → Google Sheets pipeline working
  • OLED shows name, balance, and expiry
  • Card expiry validation implemented
  • Transaction log appended to Sheets correctly
  • HTTPS communication from ESP32 to cloud verified
Led To V2 Because
  • A real seller needs a web panel, not a serial console
  • Transaction receipts are needed for accountability
  • Seller login required for security

V2
Seller Web Panel + Email Receipts
PHASE 2 · WEEKS 5–6

Version 2 introduced a complete Seller Web Panel hosted directly from Google Apps Script as an HTML web app. When a card is scanned, the ESP32 sends the UID to the server, which stores it in Script Properties. The seller's browser polls this endpoint every 2 seconds — automatically detecting the scanned card and populating the user details. The seller can then enter an amount and click Deduct or Recharge. On transaction completion, Google's MailApp sends an email receipt to the student's registered email address.

Limitations Found
  • UID polling via Script Properties has ~2s lag
  • No live feedback to OLED after transaction
  • Google Apps Script has daily email/execution quotas
  • Sheets as a database becomes slow with many rows
  • No student self-service — students can't check balance themselves
What Was Added
  • Full Seller Web Panel (login, scan, deduct, recharge, register)
  • Email receipts via MailApp after every transaction
  • Auto card detection via UID polling in browser
  • User registration and update from the panel
  • CIN and email fields added to student records
Led To V3 Because
  • OLED didn't confirm successful transactions
  • Epoch-based filtering needed to avoid showing old data
  • Insufficient balance should show on OLED immediately

V3
Real-time OLED Transaction Confirmation
PHASE 3 · WEEKS 6–7

V3 introduced a critical improvement: the ESP32 now polls for the transaction result after sending the card UID, using a Unix epoch timestamp to filter out any older transactions. Within the 15-second processing window, the OLED updates live — showing "Deducted: ₹60 / New Bal: ₹390" or "Recharge OK / New Exp: 2026-05-05" as soon as the seller confirms on the web panel. An "INSUFFICIENT BALANCE" flag stored in Script Properties is also polled and shown on the OLED immediately.

Limitations Found
  • Google Sheets still slow for large datasets
  • No real-time push — still relies on polling
  • No admin panel for managing sellers or system
  • No student portal for self-service balance checks
  • Still dependent on Google infrastructure (quotas)
What Was Added
  • Epoch-based transaction filtering (getScanEpoch)
  • OLED live confirmation of deduct/recharge result
  • Insufficient balance shown on OLED instantly
  • Recharge ≥ ₹1500 auto-extends card expiry by 30 days
  • Cleaner HTML seller panel with toasts and spinners
Led To V4 Because
  • Google Sheets is not a real database — needs Firebase
  • Admin panel and student portal are missing
  • Real-time data sync requires a proper real-time database

Firebase Architecture + Multi-Portal System
PHASE 4 · WEEKS 7–8

Version 4 was a complete architectural rebuild. Google Sheets was replaced with Firebase Realtime Database — giving true real-time synchronisation, structured security rules, and no execution quotas. Three web portals were developed: the Seller Panel (deduct, recharge, register users), the Student Portal (balance, transaction history, notices), and the Admin Panel (manage sellers, items, users, broadcast notices). V1 — Arduino Firmware

BASIC RFID → GOOGLE SHEETS · SERIAL MONITOR AMOUNT ENTRY

Algorithm — V1 Arduino Firmware
  1. Include libraries: WiFi, HTTPClient, WiFiClientSecure, SPI, MFRC522, Wire, Adafruit GFX/SSD1306, ArduinoJson, time.h
  2. Define WiFi credentials, Google Apps Script URL, RC522 pins (SS=21, RST=22), OLED pins (SDA=4, SCL=5), and NTP server
  3. setup(): Initialise Serial (115200 baud), I2C bus for OLED, SPI bus for RFID; connect to WiFi; sync time via NTP; show "System Ready"
  4. loop(): Poll MFRC522 for new card — return immediately if none present
  5. On card detected: build UID hex string (uppercase); show "Card Detected" + UID on OLED
  6. HTTP GET to Apps Script with ?uid=UID — parse JSON response
  7. If status ≠ OK: show "CARD NOT FOUND" on OLED and return
  8. Extract name, balance, expiry from JSON; strip time portion from expiry string
  9. Compare current date (from NTP) against expiry; show "CARD EXPIRED" and return if past
  10. Display student name, balance, and expiry on OLED for 3 seconds
  11. Enter 15-second window: show countdown, poll Serial.available() each 300ms for operator input
  12. If no Serial input within 15s: show "TRANSACTION CANCELLED" and return
  13. Read integer amount from Serial; validate: reject if ≤ 0 or > balance
  14. HTTP GET to Apps Script with ?action=deduct&uid=UID&amount=AMOUNT — process deduction
  15. Show "SUCCESS / Deducted: X / New Bal: Y" on OLED for 4 seconds, then reset to idle

#include <HTTPClient.h>

#include <Adafruit_GFX.h>
#include <Adafruit_SSD1306.h>

const char* password = "YOUR_WIFI_PASSWORD_HERE";

// ── Google Apps Script URL ────────────────────
String serverURL = "YOUR_WEB_APP_URL_HERE";

// ── RC522 Pins ────────────────────────────────
#define SS_PIN  21
#define RST_PIN 22
MFRC522 mfrc522(SS_PIN, RST_PIN);

// ── OLED ──────────────────────────────────────

#define SCREEN_HEIGHT 64
#define OLED_SDA     4
#define OLED_SCL     5
#define OLED_ADDR    0x3C

// ── NTP (IST = UTC+5:30) ──────────────────────
const char* ntpServer        = "pool.ntp.org";
const long  gmtOffset_sec    = 19800;
const int   daylightOffset_sec = 0;

// ── Utility: Show text on OLED ─────────────────
void showText(String l1="", String l2="", String l3="", String l4="") {
  display.clearDisplay();
  display.setTextSize(1);
  display.setTextColor(SSD1306_WHITE);
  display.setCursor(0, 0);
  display.println(l1); display.println(l2);
  display.println(l3); display.println(l4);
  display.display();
}

// ── Utility: Get current date (YYYY-MM-DD) ─────
String getCurrentDate() {
  struct tm timeinfo;
  if (!getLocalTime(&timeinfo)) return "";
  char buffer[11];
  strftime(buffer, sizeof(buffer), "%Y-%m-%d", &timeinfo);
  return String(buffer);
}

// ── Setup ──────────────────────────────────────
void setup() {
  Serial.begin(115200);
  Wire.begin(OLED_SDA, OLED_SCL);
  display.begin(SSD1306_SWITCHCAPVCC, OLED_ADDR);
  SPI.begin(18, 19, 23, 21);
  mfrc522.PCD_Init();

  showText("Connecting WiFi...");
  WiFi.begin(ssid, password);
  while (WiFi.status() != WL_CONNECTED) { delay(500); }

  configTime(gmtOffset_sec, daylightOffset_sec, ntpServer);
  showText("System Ready", "Scan Card");
  delay(2000);
}

// ── Loop ──────────────────────────────────────
void loop() {
  if (!mfrc522.PICC_IsNewCardPresent()) return;
  if (!mfrc522.PICC_ReadCardSerial())  return;

  // ── Read UID ──
  String uid = "";
  for (byte i = 0; i < mfrc522.uid.size; i++) {
    if (mfrc522.uid.uidByte[i] < 0x10) uid += "0";
    uid += String(mfrc522.uid.uidByte[i], HEX);
  }
  uid.toUpperCase();
  showText("Card Detected", uid);
  delay(1000);

  WiFiClientSecure client;
  client.setInsecure();
  HTTPClient http;

  // ── Fetch student data from Google Sheets ──
  http.begin(client, serverURL + "?uid=" + uid);
  http.setFollowRedirects(HTTPC_FORCE_FOLLOW_REDIRECTS);
  int httpCode = http.GET();
  if (httpCode <= 0) { showText("HTTP ERROR"); delay(3000); return; }

  String payload = http.getString();
  http.end();

  DynamicJsonDocument doc(512);
  deserializeJson(doc, payload);

  if (doc["status"] != "OK") {
    showText("CARD NOT FOUND"); delay(3000); return;
  }

  String name      = doc["name"];
  int    balance   = doc["balance"];
  String expiryRaw = doc["expiry"];
  String expiry    = expiryRaw.substring(0, 10);

  // ── Expiry check ──
  if (getCurrentDate() > expiry) {
    showText("CARD EXPIRED"); delay(3000); return;
  }

  showText("Name: " + name,
           "Balance: " + String(balance),
           "Expiry: " + expiry);
  delay(3000);

  // ── 15-second Serial Monitor window ──
  unsigned long startTime = millis();
  bool amountEntered = false;
  int  amount = 0;

  while (millis() - startTime < 15000) {
    int remaining = 15 - ((millis() - startTime) / 1000);
    showText("Processing...", "Please Wait",
             "Time Left: " + String(remaining) + "s");
    if (Serial.available()) {
      amount = Serial.parseInt();
      Serial.readStringUntil('\n');
      amountEntered = true; break;
    }
    delay(300);
  }

  if (!amountEntered)  { showText("TRANSACTION", "CANCELLED"); delay(3000); showText("Scan Next Card"); return; }

function doGet(e) {

  var ss        = SpreadsheetApp.getActiveSpreadsheet();
  var sheet     = ss.getSheetByName("CardHolders");
  var logSheet  = ss.getSheetByName("Transactions");

  // Create Transactions sheet if it doesn't exist
  if (!logSheet) {
    logSheet = ss.insertSheet("Transactions");
    logSheet.appendRow(["Date","Time","UID","Name","Amount","BalanceAfter"]);
  }

  var uid    = e.parameter.uid;
  var action = e.parameter.action;
  var amount = e.parameter.amount;

  if (!uid) return jsonResponse({ status: "NOT_FOUND" });

  var data = sheet.getDataRange().getValues();

  for (var i = 1; i < data.length; i++) {

    if (data[i][0].toString().trim().toUpperCase() == uid.toUpperCase()) {

      var name    = data[i][1];
      var balance = Number(data[i][2]);
      var expiry  = data[i][3];

      // Expiry check
      var today = new Date(); today.setHours(0,0,0,0);
      var expiryDate = new Date(expiry); expiryDate.setHours(0,0,0,0);
      if (expiryDate < today) return jsonResponse({ status: "EXPIRED" });

      // Fetch details (no action)
      if (!action) {
        return jsonResponse({ status: "OK", name: name, balance: balance, expiry: expiry });
      }

      // Deduct action
      if (action == "deduct") {
        amount = Number(amount);
        if (!amount || amount <= 0) return jsonResponse({ status: "INVALID_AMOUNT" });
        if (amount > balance)         return jsonResponse({ status: "INSUFFICIENT" });

        var newBalance = balance - amount;
        var now = new Date();

        // Update CardHolders sheet
        sheet.getRange(i + 1, 3).setValue(newBalance);
        sheet.getRange(i + 1, 5).setValue(now);
        sheet.getRange(i + 1, 6).setValue(now);
        sheet.getRange(i + 1, 7).setValue(amount);

        // Append to Transactions log
        logSheet.appendRow([
          Utilities.formatDate(now, Session.getScriptTimeZone(), "yyyy-MM-dd"),
          Utilities.formatDate(now, Session.getScriptTimeZone(), "HH:mm:ss"),
          uid, name, amount, newBalance
        ]);

        return jsonResponse({ status: "SUCCESS", deducted: amount, newBalance: newBalance });
      }
    }
  }

  return jsonResponse({ status: "NOT_FOUND" });
}

function handleRequest(e) {
  const ss           = SpreadsheetApp.getActiveSpreadsheet();
  const users        = ss.getSheetByName("CardHolders");
  const transactions = ss.getSheetByName("Transactions");
  const sellers      = ss.getSheetByName("Sellers");

  if (action == "cardScanned") {
    PropertiesService.getScriptProperties().setProperty("LAST_UID", e.parameter.uid);
    return json({ status: "UID_STORED" });
  }

  if (action == "getLastUID") {
    const uid = PropertiesService.getScriptProperties().getProperty("LAST_UID");
    return json({ uid: uid || "" });
  }

  if (action == "sellerLogin") {
    const data = sellers.getDataRange().getValues();
    for (let i = 1; i < data.length; i++) {
      if (data[i][0] == e.parameter.username && data[i][1] == e.parameter.password)
        return json({ status: "SUCCESS" });
    }
    return json({ status: "INVALID" });
  }

  if (action == "register") {
    const expiry = new Date(); expiry.setDate(expiry.getDate() + 30);
    users.appendRow([e.parameter.uid, e.parameter.name,
      parseFloat(e.parameter.balance), e.parameter.cin, expiry, e.parameter.email]);
    return json({ status: "REGISTERED" });
  }

  if (action == "update") {
    const data = users.getDataRange().getValues();
    for (let i = 1; i < data.length; i++) {
      if (data[i][0] == e.parameter.uid) {
        users.getRange(i+1,2).setValue(e.parameter.name);
        users.getRange(i+1,3).setValue(parseFloat(e.parameter.balance));
        users.getRange(i+1,4).setValue(e.parameter.cin);
        users.getRange(i+1,6).setValue(e.parameter.email);
        return json({ status: "UPDATED" });
      }
    }
    return json({ status: "NOT_FOUND" });
  }

  if (action == "fetchUser") {
    const data = users.getDataRange().getValues();
    for (let i = 1; i < data.length; i++) {
      if (data[i][0] == e.parameter.uid)
        return json({ status: "FOUND", name: data[i][1], balance: data[i][2],
                       cin: data[i][3], expiry: data[i][4], email: data[i][5] });
    }
    return json({ status: "NOT_FOUND" });
  }

  if (action == "deduct")   return processTransaction(e, "DEDUCT");
  if (action == "recharge") return processTransaction(e, "RECHARGE");

  return json({ status: "INVALID_ACTION" });
}

function processTransaction(e, type) {
  const ss           = SpreadsheetApp.getActiveSpreadsheet();
  const users        = ss.getSheetByName("CardHolders");
  const transactions = ss.getSheetByName("Transactions");
  const uid    = e.parameter.uid;
  const amount = parseFloat(e.parameter.amount);
  const item   = e.parameter.item;
  const data   = users.getDataRange().getValues();

  for (let i = 1; i < data.length; i++) {
    if (data[i][0] == uid) {
      let balance = parseFloat(data[i][2]);
      if (type == "DEDUCT") {
        if (balance < amount) return json({ status: "INSUFFICIENT_BALANCE" });
        balance -= amount;
      } else {
        balance += amount;
      }
      users.getRange(i+1, 3).setValue(balance);
      transactions.appendRow([new Date(), uid, data[i][1], type, item, amount, balance]);

      // Email receipt
      MailApp.sendEmail({
        to: data[i][5],
        subject: "Smart Card Transaction Receipt",
        htmlBody: "<h3>Transaction Details</h3>" +
          "Name: " + data[i][1] + "<br>" +
          "Type: " + type + "<br>" +
          "Item: " + item + "<br>" +
          "Amount: " + amount + "<br>" +
          "New Balance: " + balance
      });
      return json({ status: "SUCCESS", newBalance: balance });
    }
  }
  return json({ status: "NOT_FOUND" });
}

function handleRequest(e) {
  const ss           = SpreadsheetApp.getActiveSpreadsheet();
  const users        = ss.getSheetByName("CardHolders");
  const transactions = ss.getSheetByName("Transactions");
  const sellers      = ss.getSheetByName("Sellers");
  const action       = e.parameter.action;

  if (action == "cardScanned") {
    PropertiesService.getScriptProperties()
      .setProperty("LAST_UID", e.parameter.uid);
    return json({ status: "UID_STORED" });
  }

  if (action == "getLastTransaction") {
    const uid   = e.parameter.uid;
    const since = e.parameter.since ? new Date(parseInt(e.parameter.since)) : null;
    const flag  = PropertiesService.getScriptProperties()
                    .getProperty("LAST_TX_" + uid);
    if (flag == "INSUFFICIENT_BALANCE") {
      PropertiesService.getScriptProperties().deleteProperty("LAST_TX_" + uid);
      return json({ status: "INSUFFICIENT_BALANCE" });
    }
    const tData = transactions.getDataRange().getValues();
    for (let i = tData.length - 1; i > 0; i--) {
      if (tData[i][1] == uid) {
        if (since && new Date(tData[i][0]) <= since) return json({ status: "NONE" });
        return json({ type: tData[i][3], amount: tData[i][5], balance: tData[i][6] });
      }
    }
    return json({ status: "NONE" });
  }

  if (action == "deduct")   return processTransaction(e, "DEDUCT");
  if (action == "recharge") return processTransaction(e, "RECHARGE");

  return json({ status: "INVALID_ACTION" });
}

function processTransaction(e, type) {
  const users        = SpreadsheetApp.getActiveSpreadsheet().getSheetByName("CardHolders");
  const transactions = SpreadsheetApp.getActiveSpreadsheet().getSheetByName("Transactions");
  const uid    = e.parameter.uid;
  const amount = parseFloat(e.parameter.amount);
  const item   = e.parameter.item;
  const data   = users.getDataRange().getValues();

  for (let i = 1; i < data.length; i++) {
    if (data[i][0] == uid) {
      let balance = parseFloat(data[i][2]);
      let expiry  = new Date(data[i][4]);

      if (type == "DEDUCT") {
        if (balance < amount) {
          PropertiesService.getScriptProperties()
            .setProperty("LAST_TX_" + uid, "INSUFFICIENT_BALANCE");
          return json({ status: "INSUFFICIENT_BALANCE" });
        }
        balance -= amount;
      } else {
        balance += amount;
        if (amount >= 1500) { expiry.setDate(expiry.getDate() + 30); }
      }

      users.getRange(i + 1, 3).setValue(balance);
      transactions.appendRow([new Date(), uid, data[i][1], type, item, amount, balance]);

      MailApp.sendEmail({
        to: data[i][5],
        subject: "Smart Card Transaction Receipt",
        htmlBody: "<h3>Transaction Details</h3>" +
                  "Name: " + data[i][1] + "<br>Type: " + type +
                  "<br>Amount: " + amount + "<br>New Balance: " + balance
      });
      return json({ status: "SUCCESS", newBalance: balance });
    }
  }
  return json({ status: "NOT_FOUND" });
}

  return ContentService
    .createTextOutput(JSON.stringify(obj))
    .setMimeType(ContentService.MimeType.JSON);

V4 — Firebase Architecture

FIREBASE REALTIME DB · MULTI-PORTAL · SECURITY RULES

V4 replaces the Google Sheets/Apps Script backend entirely with Firebase Realtime Database. All four panels (Arduino firmware, Seller, Student, Admin) communicate directly with Firebase over its REST API using a legacy database secret. The section below documents the database schema and security rule logic — the complete source files are on Google Drive.

Database Structure

The Realtime Database is organised into eight top-level nodes. Each node's shape is shown below with representative sample data.

Node — /users/{rfid_uid}
  1. name — Student's full name (string, non-empty)
  2. cin — College identification number, e.g. 22BCS001 (string, non-empty)
  3. email — Used for transaction receipts (string)
  4. password — Student portal login credential (string, writable by student session)
  5. balance — Current wallet balance in ₹ (number ≥ 0, validated server-side)
  6. dailyLimit — Maximum spend per day in ₹ (number > 0)
  7. expiry — Card validity date in yyyy-MM-dd format (validated by regex rule)
Node — /transactions/{rfid_uid}/{push_id}
  1. type — Either "DEDUCT" or "RECHARGE" (enforced by security rule enum check)
  2. amount — Transaction amount in ₹ (number > 0)
  3. balance — Balance after transaction (number ≥ 0)
  4. item — Item name for DEDUCT transactions, e.g. "Veg Thali" (string, optional)
  5. sellerID — ID of the seller counter that processed the transaction (string, non-empty)
  6. timestamp — Unix epoch in milliseconds (number); used by Arduino to filter stale results
  7. Security rule prevents deletion — new data must exist or old data must not have existed (append-only ledger)
Node — /devices/{deviceId}
  1. wifiConfig — Object with ssid and password; written by Admin, read by Arduino on boot to join the correct network
  2. lastUID — Most recently scanned RFID UID; written by Arduino, polled by Seller Panel for auto-detection
  3. cancelFlag — Boolean written by Seller Panel; Arduino polls this and aborts the transaction if true
  4. lastTransaction — Object written by Seller after processing: type, amount, newBalance, timestamp, status, expiry — Arduino polls this within the 15-second window to show OLED confirmation
  5. lastSeen — Unix epoch written by Arduino on each loop to indicate the device is online
Nodes — /sellers, /admins, /items, /notices, /about
  1. /sellers/{sellerID} — name, password (≥ 4 chars), addedAt timestamp; Admin creates, Seller reads own node to verify login
  2. /admins/{adminID} — password, addedAt, isSuperAdmin boolean; used for Admin Panel login check
  3. /items/{itemId} — name, price (number ≥ 0), category; Admin manages, Seller/Student read for dropdown menus
  4. /notices/{noticeId} — message, postedBy, postedAt timestamp; Admin writes, students and sellers read for notice board
  5. /about — creator object (name, role, bio, linkedIn, photoURL) and contributors map; Admin writes, all panels read for the creator FAB

Security Rules Algorithm

All four panels authenticate using the Firebase legacy database secret (treated as a single privileged credential). Granular per-user writes are enforced through field-level .validate rules. A catch-all $other node denies all unauthenticated access.

Algorithm — Firebase Security Rules Logic
  1. All nodes require auth != null for both read and write — anonymous access is completely blocked
  2. /users: any authenticated caller may read/write; field-level validators enforce balance ≥ 0, dailyLimit > 0, expiry matches /^\d{4}-\d{2}-\d{2}$/, name and cin are non-empty strings
  3. /transactions: append-only enforced — write rule is newData.exists() || !data.exists(), so existing records cannot be deleted or overwritten; type is enum-checked to only allow "DEDUCT" or "RECHARGE"; amount must be > 0
  4. /devices: wifiConfig validated to have non-empty ssid; cancelFlag must be boolean; lastTransaction validated to have type/amount/timestamp when present, or allowed to be null (cleared after OLED confirmation)
  5. /sellers and /admins: password must be a string of at least 4 characters; addedAt must be a number
  6. /items: name non-empty, price ≥ 0, category non-empty — prevents malformed menu data
  7. /notices: message non-empty, postedBy and postedAt required — prevents blank notices
  8. $other: catch-all rule sets both .read and .write to false — any node not explicitly listed is completely inaccessible
  9. Migration path noted in rules: once student panels migrate to Firebase Auth (uid-based sessions), the student password field write can be scoped to auth.uid === $uid for true per-user isolation

V4 Arduino Firmware Algorithm

The V4 firmware replaces all Google Sheets HTTP calls with direct Firebase REST calls. WiFi credentials are fetched from /devices/canteen/wifiConfig at boot, with a hardcoded fallback. The polling loop, cancel flag, and OLED feedback pattern are retained from V3 but now use Firebase paths.

Algorithm — V4 Arduino Firmware
  1. On boot: initialise Serial, I2C (OLED), SPI (RFID), show "Smart Card V4" splash
  2. Attempt to fetch /devices/canteen/wifiConfig from Firebase using hardcoded fallback credentials; if fetch succeeds, reconnect with cloud-sourced SSID/password; otherwise continue with fallback
  3. Sync NTP time (IST, UTC+5:30); display "Ready — Tap Card" on OLED
  4. Every 30 seconds: re-check WiFi connection and reconnect if dropped (background keep-alive)
  5. On RFID card tap: read 4-byte UID, format as uppercase hex string (e.g. A1B2C3D4)
  6. GET /users/{uid}.json from Firebase — parse name, balance, expiry, dailyLimit from JSON response
  7. If user not found: display "Unknown Card" on OLED, halt
  8. Check expiry against today's date (yyyy-MM-dd); if expired, display "Card Expired" on OLED, halt
  9. Display student name, balance, expiry on OLED; record scanEpochMs (current Unix timestamp in ms)
  10. PATCH /devices/canteen/lastUID with the scanned UID — triggers Seller Panel auto-detection
  11. Enter 15-second polling loop (500 ms interval): GET /devices/canteen/lastTransaction; if timestamp > scanEpochMs, transaction is new — break loop
  12. Also poll /devices/canteen/cancelFlag in the same loop; if true, display "Cancelled" on OLED and exit
  13. On receiving new transaction: parse type, amount, newBalance, status, expiry; display result on OLED ("Deducted ₹X / Bal: ₹Y" or "Recharged ₹X / Exp: date" or "INSUFFICIENT BALANCE")
  14. Reset cancelFlag to false via PATCH; wait 4 seconds; return to "Ready — Tap Card"

V5 — PWA Transformation

PROGRESSIVE WEB APP · SERVICE WORKER · MANIFEST · GITHUB PAGES

V5 takes the V4 portals and makes them installable as native-like applications on any device — Android, iOS, or desktop — using the Progressive Web App standard. No app store required. The transformation requires three new files and small additions to each HTML portal. The complete source is on Google Drive.

Algorithm — Step 1: Project Structure Setup
  1. Organise the project folder: place admin_fixed.html, seller_fixed.html, student_fixed.html at the root
  2. Create an assets/ subfolder; place two PNG app icons inside: icon-192.png (192×192 px) and icon-512.png (512×512 px)
  3. Two new files must be created at root level: manifest.json and sw.js
  4. HTTPS hosting is mandatory — PWAs only work in a secure context; GitHub Pages (used for this project) provides HTTPS automatically
Algorithm — Step 2: Web App Manifest (manifest.json)
  1. Set name to "Smart Card System" and short_name to "SmartCard" (shown under the home-screen icon)
  2. Set start_url to ./student_fixed.html — this is the entry point when the installed app is launched
  3. Set display to "standalone" — hides the browser URL bar so the app looks native
  4. Set theme_color and background_color to #2563eb / #f8fafc to match the portal brand colour
  5. Declare both icon sizes in the icons array with purpose: "any maskable" so Android adaptive icons work correctly
  6. Set orientation to "portrait" to lock the app layout for mobile canteen use
Algorithm — Step 3: Service Worker (sw.js) — Network-First Strategy
  1. Define CACHE_NAME = 'smartcard-v4-cache-v1' and list all app shell files in ASSETS_TO_CACHE: the three HTML portals, manifest.json, and both icon files
  2. Install event: open the named cache, call cache.addAll(ASSETS_TO_CACHE) to pre-cache all shell files; call self.skipWaiting() to activate immediately without waiting for old SW to die
  3. Activate event: iterate all existing cache keys; delete any cache whose name does not match CACHE_NAME — this cleans up stale caches from previous deployments; call self.clients.claim() to take control of all open tabs immediately
  4. Fetch event: attempt network fetch first; if the network request succeeds, return the live response (ensures Firebase data is always fresh)
  5. If the network fetch fails (user is offline), fall back to caches.match(event.request) — returns the cached shell so the portal UI loads even without connectivity
  6. To update the cache in a future deployment, increment CACHE_NAME (e.g. to v2) — the activate event will automatically purge the old cache on the next visit
Algorithm — Step 4: Link PWA Files to Each HTML Portal
  1. In the <head> of each portal HTML: add <link rel="manifest" href="manifest.json"> to register the manifest
  2. Add <meta name="theme-color" content="#2563eb"> to colour the browser chrome on Android
  3. Add <link rel="apple-touch-icon" href="assets/icon-192.png"> for iOS home-screen icon support
  4. Add <meta name="apple-mobile-web-app-capable" content="yes"> to enable standalone mode on Safari/iOS
  5. Just before the closing </body> tag of each portal: add a script that checks for 'serviceWorker' in navigator and registers ./sw.js on the window load event
Algorithm — Step 5: Deployment & Installation Flow
  1. Upload the completed project folder to GitHub Pages (or any HTTPS host such as Firebase Hosting or Vercel)
  2. Android (Chrome): an "Add to Home screen" banner appears automatically at the bottom of the screen after the manifest and service worker are detected by the browser
  3. iOS (Safari): user taps the Share button → selects "Add to Home Screen" → confirms; the app icon appears on the home screen and launches in standalone mode
  4. Desktop (Chrome / Edge): an install icon (monitor with a down-arrow) appears in the far right of the URL bar; clicking it installs the app as a desktop application with its own window
  5. Once installed, the app shell (HTML, CSS, JS) loads instantly from the service worker cache; only Firebase data fetches require an active network connection
